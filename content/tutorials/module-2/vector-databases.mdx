## The problem

You embedded **10 million** chunks (1536-d float vectors). A user query becomes another vector. You need the **top-k** nearest neighbors in milliseconds — not a Python `for` loop over NumPy rows.

That is what **vector databases** (or vector **extensions** to existing DBs) optimize: **storage + ANN index + query API**.

## Core capabilities

1. **Vector index** — HNSW, IVF, diskANN variants trade RAM, build time, recall, and QPS.
2. **Metadata filters** — `WHERE team_id = 7 AND region = 'EU'` **before** or **during** vector search (vendor-specific semantics — read the docs).
3. **Hybrid search** — combine BM25-style keyword hits with embedding scores; often beats pure semantic on mixed queries.
4. **CRUD** — embeddings change when documents change; indexes must **update incrementally** or rebuild.

## HNSW in one paragraph

**Hierarchical Navigable Small World (HNSW)** builds a multi-layer graph. Upper layers are sparse “express highways” for coarse navigation; lower layers are dense for precise neighbors. A query greedy-walks down the layers — much fewer distance computations than brute force.

## 2026-ish landscape (patterns, not stock picks)

- **Dedicated SaaS** — fully managed ANN + ops (good when you want zero index tuning time).
- **Open-source servers** — self-host for control and data residency.
- **pgvector** — keep vectors beside relational rows; fantastic default for **≤ few million** vectors on a well-sized Postgres instance with tuned `lists` / `probes` or HNSW params.

```sql
-- Illustrative pgvector-style query (exact syntax varies by install)
SELECT id, content
FROM doc_chunks
ORDER BY embedding <=> $1::vector  -- distance operator
LIMIT 8;
```

<Callout type="insight">
Most “vector database” debates are really **index algorithm + filter semantics + ops story** debates. Start simple: **one Postgres** with pgvector and good monitoring; split out when QPS, recall SLAs, or multi-tenant isolation force your hand.
</Callout>

## What you still must engineer

- **Chunking** — retrieval quality lives or dies on chunk boundaries.
- **Re-ranking** — retrieve 50 with ANN, **re-rank** to 8 with a cross-encoder or lightweight model.
- **Evals** — nDCG, MRR, task success on labeled question sets.

---

## Key takeaways

- Vector DBs = **embedding storage + fast approximate nearest neighbors + filters**.
- **HNSW** is the default mental model for modern ANN graphs.
- **pgvector** is enough for many products; dedicated systems win at extreme scale or specialized SLAs.

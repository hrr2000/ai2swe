## BERT (Encoder-only)

**Pretraining task:** Masked Language Modeling (MLM) — hide random tokens, predict them using **both sides** of context. Also next-sentence prediction in original paper (later often dropped).

**Strengths:** sentence embeddings, NER, retrieval encoders, understanding benchmarks.

**Weakness:** not natively a **generator** — you add task heads or decode heuristically.

## GPT (Decoder-only)

**Pretraining task:** predict **next token** left-to-right with causal mask.

**Strengths:** coherent long-form generation, chat, code completion.

**Weakness:** no built-in bidirectional token representation in one forward pass — though **prefix** context is fully visible to later prefix tokens (left context only).

## When to use which pattern today

- Need **rich sentence encoder** for search? Fine-tune **encoder** or use modern **E5/BGE** (often encoder or encoder-decoder hybrids).
- Need **assistant** that writes? **Decoder-only** ecosystem (tooling, RLHF, APIs) is largest.

<Callout type="tip">
Many “GPT” APIs internally use **system + tool** scaffolding — architecture family name ≠ product feature set.
</Callout>

## Key takeaways

- **BERT** ≈ bidirectional denoising autoencoder for text.
- **GPT** ≈ causal autoregressive LM.
- Product choices follow **task + ecosystem**, not religion.

## Autoregressive cost without cache

Generating token `t` naively recomputes attention for positions `1…t` every step → **O(T²)** work across the whole generation.

## With KV cache

Store, per layer and head, tensors `K_{1:t}`, `V_{1:t}`. New token only computes `q_{t+1}` against cached `K,V` and **appends** its own `k_{t+1}, v_{t+1}`.

Per-step cost ~ **O(t)** attention against growing cache — still heavy for huge contexts but far better than quadratic recomputation.

## Memory footprint

KV cache RAM scales roughly:

```text
2 × layers × batch × heads × seq_len × head_dim × bytes_per_element
```

**Multi-query attention (MQA)** and **grouped-query (GQA)** shrink cache by sharing K/V heads — dominant in long-context deployments.

<Callout type="tip">
When sizing GPUs for serving, budget **KV cache + activations**, not weights alone — long concurrent chats explode memory.
</Callout>

---

## Key takeaways

- KV cache avoids **recomputing** past keys/values during decode.
- Memory grows **linearly** with sequence length × batch × layers.
- **MQA/GQA** are architectural responses to KV RAM pressure.

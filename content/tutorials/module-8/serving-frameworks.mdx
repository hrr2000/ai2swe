## Responsibilities

- **Continuous batching** — pack in-flight decode steps from many users into one GPU wavefront.
- **PagedAttention** — virtualize KV cache like OS paging to reduce fragmentation.
- **Autoscaling metrics** — GPU util, queue depth, tokens/sec, P99 latency.

<AnalogyBox
  code="Kubernetes + Envoy in front of your microservices"
  ai="Model server — schedules GPU work, exposes gRPC/HTTP, handles backpressure"
/>

## Popular stacks

| Runtime | Notes |
| --- | --- |
| **vLLM** | Python-first, strong community, PagedAttention default |
| **TensorRT-LLM** | NVIDIA-optimized, great on H100/L40S when you can compile graphs |
| **Triton Inference Server** | Multi-model routing, C++/Python backends |
| **llama.cpp / Ollama** | CPU + Apple Silicon friendly, GGUF ecosystem |

## API concerns

Expose **OpenAI-compatible** `/v1/chat/completions` when possible — swap providers without rewriting clients.

<Callout type="insight">
Measure **time-to-first-token (TTFT)** separately from **tokens/sec** — UX cares about both; batching optimizations trade them off.
</Callout>

## Key takeaways

- Serving frameworks = **schedulers + kernels + HTTP** for LLMs.
- **Continuous batching** drives GPU efficiency at scale.
- Standardize on **OpenAI-shaped APIs** internally unless you love vendor lock-in.

## The problem RAG solves

A frozen LLM is a **snapshot of the internet-ish text** it was trained on, plus whatever fits in the prompt. It does **not** know:

- Private wikis, tickets, or customer records.
- This morning’s incident postmortem.
- The exact paragraph of your Terms of Service that legal cares about.

You could **fine-tune** weights on new documents, but that is slow, expensive, and still **stale** the moment the next edit lands.

**Retrieval-Augmented Generation (RAG)** instead keeps the model weights fixed and gives the model **fresh evidence** inside the context window every query.

## The minimal pipeline

1. **Ingest** — chunk documents, compute **embeddings**, store vectors + metadata in a vector index (Module 2).
2. **Retrieve** — embed the user query, **ANN search** for top-k chunks; optionally **re-rank** with a second model.
3. **Augment** — build a prompt: system instructions + **CONTEXT:** blocks + user question.
4. **Generate** — model writes an answer conditioned on retrieved text.

```typescript
async function ragAnswer(userQuery: string) {
  const hits = await vectorIndex.search(await embed(userQuery), { k: 6 });
  const context = hits.map((h) => `SOURCE: ${h.id}\n${h.text}`).join("\n---\n");

  return llm.chat({
    messages: [
      {
        role: "system",
        content:
          "Answer using ONLY the provided context. If insufficient, say you don't know.",
      },
      {
        role: "user",
        content: `CONTEXT:\n${context}\n\nQUESTION:\n${userQuery}`,
      },
    ],
  });
}
```

## Why RAG beats “just prompt harder”

- **Freshness** — update the index when docs change; no retrain.
- **Attribution** — return `SOURCE` ids so humans can audit answers.
- **Cost** — pay embedding + small context per query instead of giant fine-tunes.

## Failure modes (the real engineering)

- **Bad chunks** — naive 512-token splits break tables and APIs across boundaries.
- **Missing retrieval** — the right paragraph was never embedded or ranked low.
- **Confident synthesis** — the model stitches unrelated chunks into a lie unless your system prompt forbids it.

<Callout type="warning">
RAG is not “truth injection.” It is **bias the distribution** toward pasted text. You still need evals, monitoring, and often **tool calls** (SQL, HTTP) for checks the model cannot do by paraphrasing prose.
</Callout>

## Relationship to fine-tuning

Use **RAG** when knowledge changes often or must be citeable. Use **fine-tuning** (or LoRA) when you need **style**, **format**, or **private skill** that retrieval cannot supply — “how we write incidents,” “our JSON schema,” “brand voice.” Many products combine **both**.

## Key takeaways

- RAG = **retrieve → stuff context → generate**.
- Vector search is the common retrieval primitive, but **hybrid lexical + metadata filters** often win in production.
- Treat chunking, re-ranking, and abstention prompts as **first-class product work**, not plumbing afterthoughts.

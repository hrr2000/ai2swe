## APIs expect structure, not vibes

Modern endpoints accept **arrays of messages** rather than a single free-form string. Even when the wire format hides it, the model ultimately consumes **tokens** — and the tokenizer plus **chat template** decides how roles become literal text with special tokens.

<AnalogyBox
  code="HTTP: middleware sets policy, body carries payload, cookies carry session"
  ai="Chat: system sets policy, user carries ask+data, assistant slots replay history"
/>

## Roles in plain language

### `system`

Long-lived instructions: safety policy, tone, output schema (“return JSON with keys …”), tool-calling rules. Keep it **stable** across turns so the model does not have to re-learn your constraints from noise.

### `user`

The evolving task: questions, pasted logs, files (as text or attachments depending on API). If you need the model to **quote** something, put that material here (or in a tool result) rather than hiding it in system text where some models deprioritize it.

### `assistant`

Prior model outputs you want the model to treat as **ground truth context** for the next reply. Also where some tool pipelines place **synthetic** assistant messages (e.g., “Calling function X…”).

```typescript
const messages: Array<{ role: "system" | "user" | "assistant"; content: string }> = [
  { role: "system", content: "You are a senior Go reviewer. No more than 6 bullets." },
  { role: "user", content: "Review this handler for context leaks:\n\n```go\n…\n```" },
  { role: "assistant", content: "1) ctx.Done() not checked …" },
  { role: "user", content: "Patch only the ctx cancellation issue." },
];
```

## Why splitting beats one blob

- **Attention allocation** — models are trained with chat formats; roles help separate **policy** from **evidence**.
- **Caching** — some providers can cache stable **system** prefixes to cut cost/latency on long policies (verify per vendor).
- **Eval harnesses** — fixtures are easier when instructions and data are separate columns in your dataset.

<Callout type="insight">
The literal string the model sees is **`apply_chat_template(messages)`** — not your JSON array. Different families (Llama, Qwen, GPT) use different separators. For local models, **always** use the tokenizer’s official template helper.
</Callout>

## Tool and multimodal messages

Many stacks add **`tool`** / **`function`** roles or structured content blocks (image bytes, audio). The pattern stays: **ordered messages** with a schema the runtime understands.

---

## Key takeaways

- Treat prompts as **message lists**, not monoliths — roles are API design.
- **System** = durable policy; **user** = task + data; **assistant** = prior turns to condition on.
- The on-wire JSON is compiled to **tokens** — debugging means inspecting that compilation.

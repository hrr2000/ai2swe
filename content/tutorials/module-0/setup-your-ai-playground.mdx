## Why you want a local playground

Hosted APIs (Gemini, OpenAI, Anthropic, etc.) are excellent for production and for many experiments. A **local playground** still earns a place on your machine because it gives you:

- **Tight feedback loops** — no network hop for every token when you are iterating on prompts or scripts.
- **Privacy** — paste internal snippets, logs, or schemas without sending them to a third party (check your employer’s policy either way).
- **Reproducibility** — pin `model:tag` the way you pin Docker images or npm versions.
- **Offline-ish work** — once weights are pulled, you can keep going on a plane or when an API has an incident.

<Callout type="tip">
Use **hosted APIs** when you want frontier capability with zero ops. Use **local models** when you want control, cost predictability on bulk jobs, or air-gapped experimentation.
</Callout>

## Ollama: pull, run, call

**[Ollama](https://ollama.com/)** downloads model weights and exposes a small local HTTP API. Think of it as a runtime that knows how to load a GGUF (or similar) bundle and execute inference.

### Install

**macOS / Linux** — from a shell:

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

**Windows** — use the installer from [ollama.com/download](https://ollama.com/download), or if you use **winget**:

```bash
winget install Ollama.Ollama
```

After install, confirm the daemon is up (it typically listens on port **11434**).

### Pull a sensible first model

Model names encode rough size. On **16 GB** unified memory or system RAM, start with **7B–8B** class models; on **32 GB+** you can explore larger quantizations.

```bash
ollama pull qwen2.5-coder:7b
ollama run qwen2.5-coder:7b
```

The `run` command opens a REPL-style chat in the terminal — quickest way to sanity-check hardware and latency.

## Call it like any HTTP service

```javascript
const response = await fetch("http://localhost:11434/api/generate", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    model: "qwen2.5-coder:7b",
    prompt:
      "Write a TypeScript function that clamps a number between min and max. One short paragraph, then code only.",
    stream: false,
  }),
});

const data = await response.json();
console.log(data.response);
```

For chat-style payloads, `/api/chat` accepts `messages: [{ role, content }, ...]` — same mental model as cloud APIs, different base URL.

## Hardware reality check

- **RAM / VRAM** caps the largest weights you can load; quantizations (Q4, Q8) trade quality for size.
- **CPU inference** is fine for short prompts; **GPU** matters when you iterate heavily or serve others.
- **Context length** still applies — stuffing an entire monorepo into one prompt may be slow or impossible on small models.

<Callout type="caution">
“Runs locally” does not mean “runs well on every laptop.” If generation is glacial, shrink the model, shorten the context, or move that workload to a GPU box and keep Ollama there.
</Callout>

## Optional: editor and agent integration

Many editors and CLI tools can target Ollama’s OpenAI-compatible surface or native endpoints. The pattern is always: **same roles, same JSON, different `baseURL`**. That is why this lesson sits in Module 0 — it is ordinary integration work.

## Key takeaways

- Ollama is a **local inference runtime** with a simple HTTP API.
- Pick model size to match **RAM/VRAM**; start with a **7B–8B** coding model on typical dev laptops.
- Treat `model:tag` as a **versioned dependency** you pin per project.

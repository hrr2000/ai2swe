## From Rules to Examples

As a software engineer, you express behavior with explicit control flow:

```javascript
// Rule-based engineering
function detectSpam(email) {
  if (email.subject.includes("FREE MONEY")) return true;
  if (email.sender === "unknown") return true;
  return false;
}
```

That works until the world moves. Attackers obfuscate text (`F r e e M 0 n e y`), languages and encodings multiply, and product rules change weekly. You end up maintaining a fragile tree of special cases — **rule-based programming** — that never quite closes the gap.

## The Shift: Examples as the Source of Truth

In supervised learning, you still write software (data loaders, model code, evaluation), but the *behavioral specification* is often a large set of **input → desired output** pairs:

```json
[
  { "input": "Get free money now!", "label": "spam" },
  { "input": "Meeting at 9am?", "label": "ham" },
  { "input": "F r e e M 0 n e y", "label": "spam" }
]
```

Think of that file the way you think of tests and fixtures:

| Traditional stack | ML stack |
| --- | --- |
| Hand-written `if` / `switch` | **Dataset** (examples + labels) |
| Compiler / bundler | **Training job** (optimizer + loss) |
| Binary / container image | **Checkpoint** (architecture + weights) |
| Unit tests on functions | **Metrics** on held-out data (precision, BLEU, win-rate) |

<AnalogyBox
  code="You commit logic as source files; CI proves behavior with tests."
  ai="You commit behavior as data; training discovers weights that satisfy that data — within limits."

## What “Debugging” Looks Like

When production misbehaves, you trace code paths. When a model misbehaves, you often trace **data**:

- **Label noise** — wrong or ambiguous labels teach the wrong thing.
- **Coverage gaps** — the model never saw “invoice fraud that looks like a refund.”
- **Train/serve skew** — features at inference time differ from training (classic bug class).

<Callout type="insight">
If outputs are systematically wrong, assume **data or evaluation** before you assume “the network is broken.” Neural code is differentiable; bad objectives or dirty examples are just as common as bad hyperparameters.
</Callout>

## Beyond Classification

The same pattern applies far beyond spam:

- **Ranking** — pairs of `(query, document, relevance score)`.
- **Summarization** — `(article, bullet summary)`.
- **Code** — `(repo context, failing test, patch)` in instruction-tuning corpora.

The artifact changes; the idea does not: **curated data defines what “good” means.**

## Guardrails Still Exist

Data does not remove engineering judgment. You still need:

- **Schema** — consistent fields, versioning, PII handling.
- **Splits** — train / validation / test, with no leakage across them.
- **Monitoring** — drift when live traffic no longer matches training.

<Callout type="tip">
Treat your dataset like a codebase: review diffs, run linters, track provenance, and pin versions the same way you pin dependencies.
</Callout>

## Key takeaways

- **Datasets** encode intent; **training** searches for weights that realize that intent.
- Garbage in, garbage out — **data quality** is the first place to look when quality drops.
- Shipping a model is shipping **weights + architecture**, analogous to shipping a compiled artifact.

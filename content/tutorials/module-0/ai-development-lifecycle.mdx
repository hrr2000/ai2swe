## The Loop You Already Know

Most teams already run a loop like this:

1. Change code or config  
2. Run automated tests  
3. Build an artifact  
4. Deploy  
5. Watch metrics and logs  

Machine learning projects run the **same loop** — the nouns change, not the rhythm.

## Stage-by-stage mapping

### 1. Writing behavior → **Data work**

Instead of only editing functions, you spend serious time on **pipelines**: collecting examples, de-duplication, labeling, balancing classes, and documenting what each field means. That is not “data science theater” — it is how you specify requirements when behavior is learned.

### 2. Unit tests → **Offline evaluation**

You still need pass/fail gates — they just look like **metrics on held-out data**:

- Classification: precision, recall, ROC-AUC, calibration.
- Generation: human eval rubrics, LLM-as-judge (carefully), task-specific scores.

The **validation split** (and often a separate **test** split you touch rarely) is your safety net against **overfitting** — the model memorized the training set but fails on new examples, like code that only passes mocked tests.

### 3. Build / compile → **Training**

Training is an expensive, batch job that turns **data + architecture + hyperparameters** into a **checkpoint** (weights). Like a long C++ compile, you want it **reproducible**: pinned data versions, seed control, logged configs (Weights & Biases, MLflow, etc.).

### 4. Deploy → **Inference**

Inference is the **online path**: given a request, load weights (or call an API), run the forward pass, return a prediction. Latency, concurrency, GPU memory, and autoscaling matter here — classic backend engineering.

### 5. Monitor → **Quality + drift**

Production ML adds a few extra monitors:

- **Data drift** — input distributions shifted (new device, new market).
- **Concept drift** — the mapping from inputs to correct outputs changed (policy rewrite).
- **Model staleness** — the world moved; time to refresh data or retrain.

<Callout type="warning">
A model can appear “healthy” (low 500 rate) while silently degrading on business metrics. Track **task-level outcomes**, not only RPC success.
</Callout>

## Environments: dev, staging, prod

Just like you do not edit prod tables by hand, you do not train in production without guardrails:

| Environment | ML analogue |
| --- | --- |
| Local / dev | Notebook or small job on a slice of data |
| Staging | Full training config on a representative snapshot |
| Production | Scheduled retrain or human-triggered job with promotion checks |

**Canary** a new checkpoint: route a small fraction of traffic, compare metrics, then roll forward or back.

---

## Key takeaways

- ML work is still **engineering**: versioned artifacts, CI gates, staged rollout, observability.
- **Training** produces a checkpoint; **inference** serves it — separate SLAs and failure modes.
- **Evaluation splits** and **monitoring** are how you keep learned behavior honest after ship.

## Prompts are not magic strings

In normal programming you call `sort(arr, opts)`. With chat models you send a **structured list of messages**. Each message has a **role** (`system`, `user`, `assistant`, sometimes `tool`) and **content** (text or multimodal parts).

The model’s job is still a function: **messages in → token stream out**. The structure exists so policies, data, and prior turns do not blur together.

## The three roles you will use daily

### System — global policy

Use this for stable instructions: persona, safety bounds, output format (JSON schema, Markdown sections), and tool-use rules. Many providers allow only one system message at the start; treat it like **application configuration**.

### User — the task and the data

Put the actual question, file contents, logs, and user-typed text here. If you mix unrelated instructions into the same blob, the model has to guess what is “data to analyze” versus “meta instructions” — splitting across roles reduces that ambiguity.

### Assistant — prior model outputs (history)

Chat APIs are **stateless**. The server does not remember your last HTTP call. Whatever context you want — prior answers, tool results you formatted as assistant turns, chain-of-thought you asked the model to hide — you must **send again** on the next request.

```typescript
const messages = [
  { role: "system", content: "You are a careful TypeScript reviewer. Be concise." },
  { role: "user", content: "Is this function safe?\n\n```ts\nexport const pick = (o: object, k: string) => (o as any)[k];\n```" },
  { role: "assistant", content: "It bypasses typing and can read unexpected keys. Prefer a typed accessor or keyof." },
  { role: "user", content: "Rewrite it with keyof and a generic." },
];
```

<Callout type="insight">
Long conversations cost tokens and latency because you **re-linearize history** every turn. Summarization, retrieval (RAG), and trimming are engineering tasks — not model magic.
</Callout>

## Streaming, tools, and “the rest of the API”

Production integrations usually add:

- **Streaming** — SSE or chunked responses so UI can render tokens as they arrive.
- **Tool / function calling** — the model emits structured JSON that your code executes, then you append a `tool` message with the result and ask again.
- **Temperature and top-p** — sampling knobs on top of next-token prediction (covered in Module 5).

You do not need those on day one, but the **message list abstraction** is the spine all of them hang on.

## Minimal Node example (hosted API)

```typescript
import { GoogleGenerativeAI } from "@google/generative-ai";

const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY!);
const model = genAI.getGenerativeModel({
  model: "gemini-2.0-flash",
  systemInstruction: "Reply with at most three bullet points.",
});

const chat = model.startChat({ history: [] });
const r = await chat.sendMessage("Why do chat models need a system message?");
console.log(r.response.text());
```

Different SDKs name things differently (`systemInstruction` vs a `system` message), but the **idea** is identical: separate long-lived policy from the user’s current ask.

## Key takeaways

- Think in **messages**, not one giant string — roles are part of your API design.
- **System** carries policy; **user** carries task + data; **assistant** carries replayed history.
- Every vendor call is **stateless** unless **you** persist and resend conversation state.

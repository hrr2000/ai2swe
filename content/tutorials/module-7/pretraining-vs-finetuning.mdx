## Pre-training

**Objective:** next-token prediction (or denoising variants) on **trillions** of tokens.

**Output:** base checkpoint with **general competence** — expensive, months of GPU fleet time, rarely done inside product companies.

## Fine-tuning (SFT)

**Objective:** supervised demonstration data `(prompt, ideal response)` to mimic a style, tool format, or policy.

**Cost:** far smaller than pretrain — still needs GPU budget, data QA, and eval harness.

## When to choose which lever

| Need | Start with |
| --- | --- |
| New factual knowledge that changes weekly | **RAG** |
| Style / tone / output schema | **SFT or LoRA** |
| Safety preference shifts | **RLHF / DPO** (later lessons) |

<Callout type="tip">
Most teams **prompt + RAG** first, **LoRA** second, **full fine-tune** last — diminishing returns vs operational risk.
</Callout>

---

## Key takeaways

- **Pre-training** = internet-scale, general-purpose.
- **Fine-tuning** = domain/task specialization on top.
- Pick the **cheapest intervention** that moves your eval metrics.

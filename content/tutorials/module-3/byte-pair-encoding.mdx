## The problem BPE solves

Character vocab → **long** sequences. Word vocab → **huge** tables and infinite OOV. **Subword** tokenization splits the difference: frequent words stay whole; rare words decompose into learnable pieces.

## Algorithm sketch

1. Start with **symbols** = bytes or UTF-8 code units (robust to any Unicode).
2. Count **adjacent pairs** in a large corpus.
3. **Merge** the most frequent pair into a new symbol; repeat until you hit target vocabulary size `V`.
4. **Encode** new text by greedily applying the merge table longest-first (implementation details vary — SentencePiece vs GPT-style).

## Why GPT-style tokenizers differ

Training corpora, **pre-tokenization** (regex splits), and **byte-level** fallbacks change the merge statistics. That is why `cl100k_base` ≠ `llama` tokenizer — **never** mix tokenizers across a pipeline.

## Gotchas

- **Whitespace** merges into weird tokens — affects code indentation cost.
- **Case** — uncased merges share `hello` / `Hello` statistics differently than cased models.
- **Numbers** — long digit strings may explode into many single-digit tokens unless pre-split.

<Callout type="tip">
Inspect merges with `tiktoken` or the Hugging Face `tokenizer.convert_ids_to_tokens` — debugging generation bugs often starts at **token boundaries**, not characters.
</Callout>

## Key takeaways

- BPE = **data-driven merges** up to a fixed vocab size.
- Explains **subwords**, **rare word splits**, and **stable OOV** handling.
- Tokenizer choice is a **versioned asset** — store it with the checkpoint.

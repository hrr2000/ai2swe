## What lives in `vocab.json` (conceptually)

- **Token strings** → **integer IDs** (0 … V−1).
- **Special entries** — pad, unk, bos, eos, chat markers — occupy low or high IDs depending on format.
- **Unused slots** — some tables reserve space for compatibility across versions.

## Vocabulary size tradeoffs

| Larger V | Effect |
| --- | --- |
| Fewer tokens per sentence | Shorter contexts, cheaper training |
| Sparse tail | Rare tokens under-trained |
| Embedding table rows | `V × d` parameters just for input lookup |

LLMs often sit between **32k** and **256k** vocab entries — not millions like raw Unicode words.

## Byte-level BPE

Falling back to **bytes** guarantees **no unknown symbol** can appear in input — every file is representable. Tradeoff: non-Latin text may consume more tokens than a language-specific tokenizer.

<Callout type="warning">
Loading weights trained with vocab **V₁** into code that tokenizes with **V₂** silently maps words to **wrong rows** — catastrophic garbage. Pin tokenizer **hash** in model cards.
</Callout>

## Key takeaways

- Vocabulary = **legal token ID space** for a checkpoint.
- Size affects **sequence length**, **memory**, and **rare token quality**.
- Always ship **tokenizer + config** with the model — they are part of the binary interface.
